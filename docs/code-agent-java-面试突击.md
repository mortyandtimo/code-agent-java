# code-agent-java 面试突击

## 1. 项目一句话介绍

`code-agent-java` 是一个基于 `Spring Boot + MySQL + MVC + DDD + OpenAI-compatible tool-calling` 实现的代码任务智能体系统，既提供 REST API，也提供交互式 CLI，可完成“任务创建 -> 多轮工具调用 -> 结果回填 -> 会话恢复 -> 最终答复”的完整链路。

## 2. 为什么要做这个项目

这个项目的目标不是做一个普通聊天后端，而是把“大模型代理执行”拆成一个后端工程项目：

- 有明确的领域对象：任务、会话、事件日志、工具调用
- 有真实的执行链路：模型请求、工具执行、结果写回、循环继续
- 有交互入口：REST + CLI
- 有工程问题：重试、超时、危险操作确认、历史恢复、配置抽离

它更适合放在简历里作为“AI Agent 工程化落地”项目，而不是单纯的 SDK Demo。

## 3. 技术栈

- 后端框架：Spring Boot 2.7.18
- 持久化：Spring Data JPA
- 数据库：MySQL
- 迁移：Flyway
- CLI：JLine + JNA + Jansi
- 模型接入：OpenAI-compatible `chat/completions`
- 工具协议：原生 `tool_calls`
- 测试：Spring Boot Test、WebMvcTest

## 4. 项目结构

项目主要按 DDD + MVC 分层：

- `application`
  - 命令对象与应用服务
  - 负责任务编排与 agent loop 执行主逻辑
- `domain`
  - 任务、会话、事件日志、LLM、工具等领域模型与仓储接口
- `infrastructure`
  - OpenAI-compatible 网关
  - JPA 仓储实现
  - 工具执行器
  - 系统提示词构建
- `interfaces`
  - REST Controller
  - CLI Runner
  - DTO
- `shared`
  - 异常与通用错误处理

## 5. 核心执行链路

### 5.1 任务创建

用户通过 REST 或 CLI 输入一个任务请求，例如：

- 读取代码
- 修改文件
- 删除精确文件路径
- 查看目录结构
- 继续历史任务

系统会先创建：

- `AgentTask`
- `AgentSession`
- 第一条事件日志

### 5.2 Agent Loop

主循环在 `AgentApplicationService` 中完成，核心流程是：

1. 组装系统提示词 + 历史消息 + 当前用户输入
2. 调用 `OpenAiCompatibleLlmGateway`
3. 判断模型是否返回 `tool_calls`
4. 如果有工具调用，就执行本地工具
5. 把工具结果作为 `tool` 消息写回上下文
6. 继续下一轮请求，直到模型输出最终自然语言答复

这本质上就是最小可用的 agent loop。

### 5.3 工具执行

当前项目支持这些工具：

- `read_file`
- `write_file`
- `edit_file`
- `list_files`
- `delete_file`
- `run_shell`

工具层不只是“能执行”，还做了额外控制：

- 危险命令确认
- 路径标准化
- Windows 删除命令兜底
- `run_shell` 超时截断与部分结果保留

### 5.4 会话恢复

项目支持恢复历史任务：

- `/resume` 菜单选择最近任务
- `/resume <taskId>` 直接切换

恢复后会把任务摘要与最近上下文重新打印到当前窗口，便于用户确认当前操作的是哪一个任务。

## 6. 我认为这个项目的亮点

### 亮点一：不是单纯聊天，而是“任务驱动型 Agent 后端”

这个项目把模型调用落到了工程实体上，而不是只做一个 `/chat` 接口。

系统内部明确区分：

- 任务
- 会话
- 事件日志
- 工具调用

所以它更接近“可审计、可恢复、可扩展”的代理系统。

### 亮点二：CLI 交互不是简单命令拼接

CLI 里做了不少工程化交互能力：

- Slash commands
- 本地 shell 快捷执行
- 历史任务恢复
- 危险操作确认
- 流式模型输出
- 任务选择菜单

这使它不只是一个后端 API，也能直接作为本地代理终端使用。

### 亮点三：处理了很多“真实世界问题”

做这个项目时，不只是实现 happy path，还处理了不少实际坑：

- Windows CLI 中路径里的反斜杠被吞掉
- 旧会话恢复时 `tool` 历史消息结构不规范，导致模型协议报错
- `run_shell` 超时但已经拿到部分结果时，不能机械当成彻底失败
- 删除单文件路径不能依赖模型自由生成 shell 命令，否则会改坏路径

这类问题很适合在面试里讲，因为它体现的是工程调试与协议理解能力。

## 7. 可以重点讲的两个难点

### 难点一：OpenAI-compatible 历史消息重建

问题：

恢复旧任务时，历史消息不是直接字符串拼回去就行，因为：

- `assistant` 可能带 `tool_calls`
- `tool` 消息必须带 `tool_call_id`
- 某些旧历史里 `name` 缺失会导致上游 400

解决：

- 对历史消息做结构化规范化
- 针对缺失的 `tool.name` 做本地兜底恢复
- 保证再次请求时消息结构满足 OpenAI-compatible 协议要求

### 难点二：Windows 路径与 CLI 输入解析

问题：

用户输入 `G:\1\1.txt` 时，CLI 输入层会把反斜杠当 escape，结果路径在进入模型前就变成了 `G:11.txt`。

解决：

- 调整 JLine Parser，关闭该输入场景中的反斜杠转义
- 对删除单文件路径改成专用 `delete_file` 工具
- 在系统提示中明确“保留绝对路径原样，不允许改写”

这体现的是对终端输入层、模型层、工具层三层边界的理解。

## 8. 面试时可直接复述的项目亮点说法

### 版本一：简历口语版

我做了一个 Java 版代码任务智能体系统，核心是把 OpenAI-compatible 的 tool-calling 协议工程化落地到了 `Spring Boot + MySQL + DDD` 项目里。它不是普通聊天接口，而是围绕任务、会话、事件日志来做多轮执行，支持 REST 和交互式 CLI 两种入口，并且支持工具调用、任务恢复、危险操作确认和流式输出。

### 版本二：偏技术版

这个项目里我重点解决了两类问题：第一类是 agent loop 的协议问题，比如历史消息恢复时 `assistant tool_calls` 和 `tool` 消息的结构化重建；第二类是本地代理执行中的工程问题，比如 Windows 路径解析、命令超时后的部分结果保留，以及删除单文件路径不能依赖模型自由生成 shell 命令，而是要走专用工具。

## 9. 如果面试官追问“还可以怎么扩展”

可以继续往这些方向扩展：

- 增加更多工具，例如目录移动、diff、grep、git 操作
- 加用户与权限系统，把危险工具做更细粒度授权
- 增加异步任务队列，把长任务和短任务分离
- 做前端控制台页面，把任务状态和事件流可视化
- 抽象 provider 层，接多家模型服务
- 加上下文压缩与摘要机制，支持更长会话恢复

## 10. 一句话总结

这个项目最适合体现的不是“会不会调 API”，而是：

我能把一个 AI Agent 从协议、后端建模、工具执行、交互 CLI、错误恢复一路做成一个能跑、能恢复、能调试、能展示的 Java 工程项目。
